(1) Modélisation et analyse du réseau de pétri

Le réseau de pétri modélisé est représenté dans la figure 1.

Figure 1 : Modélisation du réseau de pétri

Lois de conservation

Afin de faciliter l'écriture, les marquages suivant seront représentés par :

  • M(Tunnel) = M(T)
  • M(traverse tunnel MTL) = M(ttMTL)
  • M(traverse tunnel R-S) = M(ttRS)

Le réseau de pétri respecte la loi de conservation suivante :

M(T) + M(ttMTL) + M(ttRS) = c(M), ou c(M) = 1, une constante

On regarde maintenant le marquage après chaque transition qui influe sur le nombre de jetons des places Tunnel, traverse tunnel MTL et traverse tunnel R-S.

  • Initialement : M0(T) = 1, M0(ttMTL) = 0, M0(ttRS) = 0
  • Après la transition t1 : Mt1(T) = 0, Mt1(ttMTL) = 1, Mt1(ttRS) = 0, c(Mt1) = 1
  • Après la transition t2 : Mt2(T) = 1, Mt2(ttMTL) = 0, Mt2(ttRS) = 0, c(Mt2) = 1
  • Après la transition t1' : Mt1'(T) = 0, Mt1'(ttMTL) = 0, Mt1'(ttRS) = 1, c(Mt1') = 1
  • Après la transition t2' : Mt2'(T) = 1, Mt2'(ttMTL) = 0, Mt2'(ttRS) = 0, c(Mt2') = 1

Dans tous les cas, la loi de conservation est respectée pour toutes les transitions et donc soit une seule voiture est dans le tunnel à la fois, soit aucune voiture n'est dans le tunnel.

Contrainte OCL d'expiration

Afin d'encadrer formellement la sémantique de l'expiration d'une réservation, nous avons spécifié la contrainte OCL suivante. Celle-ci stipule que si le statut courant d'une réservation est EN_ATTENTE_PAIEMENT et que le délai autorisé depuis sa création est écoulé (ex: 24h), alors invraisemblablement l'opération verifierExpiration() ou similaire doit faire passer son statut à ANNULEE. 

context Reservation::verifierExpiration()
pre: 
    self.statut = StatutReservation::EN_ATTENTE_PAIEMENT and 
    self.dateCreation.plus(DelaiMaximum) < now()
post: 
    self.statut = StatutReservation::ANNULEE

(4) Qualité du design

4.1 Identification des modules à usage général et réutilisables

Notre design a été découpé en paquets de façon à maximiser la réutilisabilité. Les trois paquets suivants sont considérés à usage général, c'est-à-dire qu'ils peuvent être intégrés dans un autre système sans modification majeure :

  • core.transport : Ce paquet regroupe toutes les abstractions liées aux transports (Terminal, Vehicule, Section, UniteReservable, Siege, Cabine, Compagnie, Admin). Il est très cohésif (toutes les classes concernent le domaine du transport) et faiblement couplé aux autres paquets : d'autres paquets dépendent de lui, mais lui ne dépend de personne. Il peut donc être réutilisé dans tout système de transport (train, avion, bateau).
  • core.paiement : Ce paquet contient Paiement, MethodePaiement et StatutPaiement. Il encapsule toute la logique de paiement indépendamment du domaine métier. Il pourrait être intégré dans n'importe quelle application nécessitant un module de paiement (commerce en ligne, location, etc.).
  • domain.voyage : Ce paquet définit les entités centrales du voyage (SegmentVoyage, Vol, Itineraire, Trajet). Il peut être réutilisé dans tout système de planification d'itinéraires multimodaux.

4.2 Couplage et cohésion

Cohésion des paquets

Tous nos paquets présentent une cohésion fonctionnelle (le meilleur type de cohésion) : chaque paquet regroupe des classes qui contribuent toutes à une même responsabilité bien délimitée.

Paquet Type de cohésion Justification
controller Fonctionnelle ✅ VoyageController et Admin gèrent exclusivement les opérations de coordination du système (façade).
core.transport Fonctionnelle ✅ Terminal, Vehicule, Section, UniteReservable — tous liés aux abstractions du transport.
core.reservation Fonctionnelle ✅ Reservation, StatutReservation, Scheduler — tous liés au cycle de vie d'une réservation.
core.paiement Fonctionnelle ✅ Paiement, MethodePaiement, StatutPaiement — tous liés au traitement du paiement.
domain.voyage Fonctionnelle ✅ SegmentVoyage, Vol, Itineraire, Trajet — tous liés à la notion de voyage.
domain.client Fonctionnelle ✅ Client et ses associations directes — paquet centré sur la gestion du client.

Cohésion des classes principales

Classe Type de cohésion Justification
VoyageController Fonctionnelle Coordonne la recherche, la réservation et le paiement — rôle de façade unique et bien défini.
Reservation Fonctionnelle Gère exclusivement l'état et les unités d'une réservation (confirmer, annuler, calculer total).

Couplage entre paquets

Le couplage observé entre nos paquets est essentiellement de type données (les meilleur type) ou empreinte, ce qui garantit une bonne indépendance entre modules.

Relation Type de couplage Justification
controller → domain Données ✅ Passe des paramètres simples (identifiants, critères de recherche) — aucun objet complexe partagé.
controller → core.reservation Données ✅ Passe des IDs et informations de paiement — échange minimal de données.
core.reservation → core.paiement Données ✅ Association simple : une Reservation est liée à un Paiement.
domain.voyage → core.transport Empreinte ✅ Utilise les classes abstraites Terminal et Vehicule — dépendance sur des types, non sur des implémentations concrètes.
domain.client → core.reservation Données ✅ Le Client possède une collection de Reservation — association de composition simple.

Conclusion : Le système présente un couplage faible (données / empreinte) et une cohésion forte (fonctionnelle) → BON DESIGN.

4.4 Graphe Instabilité / Abstraction (I/A)

Formules utilisées

  • Instabilité (I) = Ce / (Ca + Ce)  — où Ca correspond aux dépendances entrantes et Ce aux dépendances sortantes.
  • Abstraction (A) = Na / Nc  — où Na est le nombre de classes abstraites et Nc est le nombre total de classes dans le paquet.
  • Distance (D) = |A + I − 1|  — idéalement D ≈ 0 (sur la Main Sequence).

Selon le principe du SAP un paquet stable devrait être plus abstrait. Dans notre design, nous avons donc considérés les classes Compagnie, SegmentVoyage, Terminal, Vehicule, Section et UniteReservable comme étant abstraites, car elles servent de classes de base dans des hiérarchies de généralisation. Notre choix fait du sens si on se base sur les heuristiques du cours, où les classes mères sont généralement abstraites pour permettre l'extension du système sans modifier la logique.

Analyse de l'allure du graphe

En analysant notre graphe IA, on peut observer que le paquet Reservations se situe au point (1, 0) ce qui correspond à un paquet très instable et complètement concret. On peut dire que c'est logique, car il dépend de plusieurs autres modules pour fonctionner.

Le paquet Voyages a une instabilité moyenne avec un faible niveau d'abstraction, ce qui nous montre qu'il dépend partiellement des autres paquets tout en restant majoritairement concret.

Les paquets Paiements et Vehicules sont quant à eux très stables, car ils ont peu ou pas de dépendances sortantes. Par contre, leur niveau d'abstraction est faible, surtout pour Paiements qui se retrouve au point (0, 0). Cela indique que ces paquets sont plus rigides et pourraient être plus difficiles à modifier ou à faire évoluer.

Conclusion : Globalement, le design respecte partiellement le Stable Abstractions Principle, mais certains paquets stables ne sont pas assez abstraits, ce qui peut créer des limitations en termes de flexibilité et d'évolution du système.

4.5 Principes de conception appliqués

Principes SOLID

Principe Appliqué ? Où / Comment
S — Responsabilité unique ✅ Oui Chaque classe a une responsabilité unique et bien délimitée : Client gère les clients, Reservation gère les réservations, Paiement gère les paiements.
O — Ouvert/Fermé ✅ Oui Les classes abstraites (Compagnie, Terminal, Vehicule, Section) permettent d'étendre le système (ex. : nouveau type de transport) sans modifier les classes existantes.
L — Substitution de Liskov ✅ Oui Vol, Itineraire, Trajet sont substituables à SegmentVoyage ; Siege et Cabine sont substituables à UniteReservable sans altérer le comportement.
I — Ségrégation des interfaces ⚠️ Partiel Il n'y a pas d'interfaces explicites dans le design actuel, mais les classes abstraites jouent un rôle similaire. L'ajout d'interfaces (ex. : IPaiement) renforcerait ce principe.
D — Inversion des dépendances ⚠️ Partiel VoyageController dépend de classes abstraites (Compagnie, Vehicule) pour les aspects transport, mais dépend aussi de classes concrètes dans certains cas. L'introduction d'interfaces améliorerait ce point.

Principes GRASP

Principe Appliqué ? Où / Comment
Expert en information ✅ Oui Reservation calcule son total, Section calcule son propre prix — chaque classe opère sur ses propres données.
Créateur ✅ Oui Client crée des Reservation (il les possède) ; VoyageController crée les résultats de recherche.
Contrôleur ✅ Oui VoyageController est le contrôleur du système — point d'entrée unique pour les cas d'utilisation (façade).
Faible couplage ✅ Oui Les paquets sont découplés par des interfaces claires ; les dépendances vont toujours du plus instable vers le plus stable.
Forte cohésion ✅ Oui Chaque paquet a une responsabilité clairement définie (transport, paiement, réservation, voyage).
Polymorphisme ✅ Oui Les hiérarchies d'héritage (Terminal, Vehicule, SegmentVoyage, UniteReservable) permettent le polymorphisme sans conditionnelles explicites.
Fabrication pure ✅ Oui VoyageController n'est pas une entité du domaine métier — c'est une fabrication pure introduite pour coordonner les interactions sans alourdir les classes du domaine.
Indirection ✅ Oui VoyageController sert d'intermédiaire entre Client et les modules core.*, réduisant le couplage direct.
Protection des variations ✅ Oui Les classes abstraites (Vehicule, Terminal, Compagnie) protègent le reste du système contre les variations dans les types de transport concrets.
