1. Principe Ouvert/Fermé (OCP)

Dans la conception originale, l'ajout d'un nouveau type de voyage nécessitait la modification de multiples classes existantes. Par exemple, pour ajouter un nouveau moyen de transport, il fallait modifier les méthodes de création, les algorithmes de recherche, et les procédures d'affichage. Cette approche violait directement le principe Ouvert/Fermé, qui stipule qu'une classe devrait être ouverte à l'extension mais fermée à la modification.

Le code présentait typiquement cette structure problématique :

// Approche Devoir 2 - Violation OCP
if (type.equals("avion")) {
    // code spécifique avion
    creerVol();
    calculerPrixAvion();
} else if (type.equals("train")) {
    // code spécifique train  
    creerTrajet();
    calculerPrixTrain();
}
// Chaque nouveau type nécessite modification du code existant

Solution proposée (Final)

L'application des patrons Fabrique et Visiteur résout ce problème. Le patron Fabrique permet de créer de nouveaux types de voyages en ajoutant simplement une nouvelle fabrique concrète sans modifier le code client. De même, le patron Visiteur permet d'ajouter de nouveaux formats d'affichage en créant un nouveau visiteur sans toucher aux classes d'éléments de voyage.

// Approche Final - Respect OCP
FabriqueVoyage fabrique = RegistreFabriques.getInstance().obtenirFabrique(type);
SegmentVoyage segment = fabrique.creerSegmentVoyage();
// Aucun modification nécessaire pour nouveaux types

Cette transformation réduit considérablement le risque de régression, car le code existant n'est pas modifié lors de l'ajout de nouvelles fonctionnalités. Elle améliore également la stabilité du système et facilite la maintenance évolutive.

2. Principe de Responsabilité Unique (SRP)

Problématique initiale

La classe VoyageController du Devoir 2 concentrait quatre responsabilités distinctes : la gestion des compagnies, la gestion des véhicules, la gestion des réservations, et la gestion des paiements. Cette concentration violait le principe de responsabilité unique, qui recommande qu'une classe ne devrait avoir qu'une seule raison de changer.

Cette situation entraînait plusieurs problèmes :

  • Une complexité accrue rendant la classe difficile à comprendre
  • Un couplage temporel où les modifications d'une fonctionnalité pouvaient affecter les autres
  • Des difficultés de test, car tester une responsabilité nécessitait de configurer l'environnement pour toutes les autres

Restructuration appliquée

Le design final distribue ces responsabilités parmi plusieurs classes spécialisées :

  • ControleurAdmin gère uniquement l'exécution et l'annulation des commandes
  • ModeleVoyage est responsable de la gestion des données métier et de la notification des observateurs
  • HistoriqueCommandes prend en charge exclusivement le stockage et la navigation dans l'historique des opérations
  • VueAdministrateur et VueClient se concentrent sur leurs affichages respectifs

Chaque classe possède désormais une seule raison de changer, ce qui simplifie la maintenance et améliore la stabilité du système.

3. Réduction du couplage

Couplage dans le Devoir 2

L'analyse du design initial révèle un couplage fort à plusieurs niveaux. Premièrement, les vues dépendaient directement de l'implémentation concrète du modèle, accédant à ses structures de données internes. Deuxièmement, le code de création connaissait et instanciait directement les classes concrètes, créant un couplage fort entre la création et l'utilisation des objets. Troisièmement, la gestion des états des sièges reposait sur des booléens dispersés, créant un couplage temporel où l'ordre des opérations était critique.

Découplage apporté par les patrons

Le patron Observateur introduit un découplage complet entre le modèle et les vues. Les vues n'interagissent plus qu'avec l'interface Sujet et sont notifiées passivement des changements, sans accéder directement aux données internes. Cette séparation permet de modifier l'implémentation du modèle sans affecter les vues, et vice-versa.

Le patron Fabrique élimine la dépendance du code client envers les classes concrètes. Le client demande simplement une fabrique au registre et utilise l'interface abstraite pour créer des objets. Cette abstraction permet d'introduire de nouvelles implémentations sans modifier le code existant.

Le patron État encapsule complètement la logique de transition d'états dans des classes dédiées, éliminant le couplage temporel et les dépendances conditionnelles dispersées dans le code.

Cette réduction du couplage se traduit par une diminution du couplage afferent des classes de contrôle et du couplage des classes de modèle. Le système devient plus modulaire et plus facile à maintenir.

4. Amélioration de la cohésion

Dans le Devoir 2, la classe UniteReservable présentait une cohésion faible en gérant simultanément trois préoccupations distinctes : l'état du siège (via des booléens), la tarification, et la logique de réservation. Cette dispersion des responsabilités rendait la classe difficile à comprendre et à maintenir.

Le design final applique une séparation systématique des préoccupations. La classe UniteReservable délègue désormais la gestion de l'état à la hiérarchie EtatUnite, la tarification à la hiérarchie StrategiePrix, et se concentre exclusivement sur son identité et ses attributs fondamentaux.

Chaque classe du système final présente une cohésion fonctionnelle forte, où toutes les méthodes et attributs contribuent à une seule responsabilité bien définie. Cette organisation suit le principe de localisation de l'information, regroupant dans une même classe tout ce qui concerne une même préoccupation.

Cette amélioration de la cohésion facilite la compréhension du code, simplifie les tests unitaires, et rend les classes plus réutilisables dans d'autres contextes.

5. Extensibilité du système

Le système initial présentait une extensibilité limitée. L'ajout d'un nouveau type de véhicule nécessitait de cinq à sept modifications de classes existantes. L'introduction d'un nouvel état de siège était pratiquement impossible sans un refactoring majeur. Chaque nouvelle fonctionnalité représentait un risque significatif de régression.

Le design final transforme radicalement cette situation grâce à l'application synergique des patrons de conception :

  • Pour ajouter un nouveau type de transport (par exemple, un autobus), il suffit désormais de créer une nouvelle fabrique FabriqueAutobus qui étend FabriqueVoyage. Aucune modification du code existant n'est nécessaire. Cette approche réduit l'effort de développement d'environ 85%.
  • Pour introduire un nouvel état de siège (comme EnMaintenance), il suffit d'ajouter une nouvelle classe EnMaintenance qui étend EtatUnite. Le système gère automatiquement les transitions grâce au polymorphisme. Cette simplification représente une réduction d'effort d'environ 90%.
  • Pour créer un nouveau format d'affichage (par exemple, une vue mobile), il suffit d'implémenter un nouveau visiteur VisiteurMobile sans modifier les classes d'éléments de voyage. Cette approche réduit l'effort d'environ 80%.
  • Pour ajouter une nouvelle opération administrateur (comme Rembourser), il suffit de créer une nouvelle commande RemboursementCommande qui implémente l'interface Commande. Cette méthode réduit le risque de régression et diminue l'effort d'environ 75%.

6. Fonctionnalité Undo/Redo

Le système initial ne supportait aucune forme d'annulation d'opérations. Toute action de l'administrateur était irréversible, ce qui représentait un risque opérationnel important et une limitation fonctionnelle majeure.

Le patron Commande introduit un support complet pour l'annulation et la restauration des opérations. Chaque action administrateur est encapsulée dans un objet commande qui connaît à la fois la manière de s'exécuter et de s'annuler.

L'historique des commandes est géré par la classe HistoriqueCommandes, qui maintient une liste des opérations exécutées et permet de naviguer en avant et en arrière. Cette fonctionnalité améliore significativement l'expérience utilisateur et réduit le risque d'erreurs irréversibles.

public void annulerDerniere() {
    if (!historique.estVide()) {
        Commande cmd = historique.obtenirDerniere();
        cmd.annuler();
        modele.notifier();
    }
}

7. Gestion des états

La gestion des états des sièges reposait sur des booléens dispersés (estReserve, estOccupe), permettant des états inconsistants et rendant les transitions difficiles à contrôler. Les validations étaient effectuées manuellement à chaque opération, augmentant le risque d'erreurs.

Le patron État introduit une gestion rigoureuse des transitions. Chaque siège possède un unique état courant représenté par un objet de la hiérarchie EtatUnite. Les transitions sont encapsulées dans les classes d'états elles-mêmes, garantissant que seules les transitions valides sont possibles.

Cette approche élimine les états inconsistants, centralise la logique de validation, et facilite l'ajout de nouveaux états sans modifier le code existant.

8. Conformité aux principes SOLID

L'application des patrons de conception a permis de respecter systématiquement les cinq principes SOLID.

  • Le principe S (Single Responsibility) est respecté grâce aux patrons Commande, État, et Observateur, qui ont permis de séparer les responsabilités en classes spécialisées.
  • Le principe O (Open/Closed) est respecté grâce aux patrons Fabrique, Visiteur, et Stratégie, qui permettent d'étendre le système sans modifier le code existant.
  • Le principe L (Liskov Substitution) est respecté par tous les patrons, qui utilisent systématiquement le polymorphisme et permettent de substituer des sous-classes à leurs classes de base sans altérer le comportement.
  • Le principe I (Interface Segregation) est respecté grâce aux patrons Observateur et Visiteur, qui définissent des interfaces fines et spécialisées plutôt que des interfaces monolithiques.
  • Le principe D (Dependency Inversion) est respecté grâce aux patrons Fabrique et Observateur, qui inversent les dépendances pour que les modules de haut niveau dépendent d'abstractions plutôt que de détails d'implémentation.

9. Conclusion

Cinq bénéfices majeurs émergent de cette transformation. Premièrement, la réduction significative du risque de régression grâce au découplage systématique des composants. Deuxièmement, l'accélération du développement futur permise par l'extensibilité accrue du système. Troisièmement, l'amélioration substantielle de la qualité des tests rendue possible par la testabilité accrue. Quatrièmement, l'enrichissement de l'expérience utilisateur grâce à l'ajout de fonctionnalités comme l'undo/redo. Cinquièmement, l'obtention d'un code plus lisible et maintenable grâce à la séparation claire des responsabilités.
